SOLID 원칙
SRP - 단일 책임 원칙 (Single Responsibility Principle)
	하나의 클래스는 하나의 책임만 가져야 함

OCP - 개방-폐쇄 원칙 (Open/Closed Principle)
	새로운 기능이나 요구사항이 들어올 때
	기존 코드를 크게 수정하지 않고도 기능을 추가할 수 있어야 함

LSP - 리스코프 치환 원칙 (Liskov Substitution Principle)
	부모 클래스를 사용하는 모든 코드가 자식 클래스에서도 동일하게 동작해야 한다
	
ISP - 인터페이스 분리 원칙 (Interface Segregation Principle)
	인터페이스는 클라이언트가 필요로 하는 기능만 제공해야 한다.

DIP - 의존성 역전 원칙 (Dependency Inversion Principle)
	상위 모듈은 하위 모듈의 구현체가 아니라, 추상화에 의존해야 한다.


EJB(Enterprise JavaBeans)라는 복잡한 인터페이스와 라이프사이클 메서드를 강제했었음

POJO	Plain Old Java Object
종속되지 않는 일반 자바 객체
EJB나 스프링을 비롯한 프레임워크에 종속되지 않고, 어디서든 자유롭게 객체를 생성하고 사용할 수 있다

POJO 기반으로 클래스를 설계하면, 불필요한 상속이나 애노테이션 없이도
객체지향 설계 원칙을 그대로 적용할 수 있으며, 유지보수와 테스트가 쉬워진다

POJO 기반 설계는 TDD(Test-Driven Development)와 매우 잘 맞습니다.
프레임워크 설정 없이 바로 테스트를 작성하고 실행할 수 있어, 빠른 피드백 루프를 만들 수 있다

JUnit	unit test	테스트를 용이하게
TDD		Test-Driven Development 테스트 주도 개발


Spring 프레임워크는 POJO 기반 개발을 지원하는 대표적인 프레임워크
Spring은 POJO에 부가 기능을 추가할 뿐, POJO의 본질을 해치지 않는다

Spring Boot는 Spring 프레임워크를 기반으로 한 도구로,
복잡한 Spring 구성을 단순화하고 더 빠르고 쉽게 애플리케이션을 개발할 수 있도록 지원

Convention over Configuration: 개발자가 이미 알고 있다고 가정하고 기본 설정을 제공
자동 구성(Auto-configuration): 클래스패스 기반으로 애플리케이션을 자동 구성
독립 실행 가능: 내장 서버로 별도 WAS 설치 없이 실행 가능
운영 준비 완료: Actuator를 통한 모니터링 기능 제공
